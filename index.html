AUDIT-001 — Findings

We scanned every AI workflow.
Here's what we found.

Three critical issues that improve.sh is designed to solve. Every finding carries file:line evidence, impact, effort, and confidence.

F-001 · CRITICAL

Every task runs on your most expensive model

Expensive models are used for everything — including simple refactors, boilerplate generation, and tests. Intelligence doesn't compound because there's no architect. Every prompt is a greenfield generation.

⚡ High impact 🔄 Effort: M 🎯 Confidence: HIGH
F-002 · CRITICAL

No codebase context before generation

Models generate against a blank slate every time. Existing patterns, conventions, and architecture decisions are invisible. Each generation risks drifting further from the codebase's actual design.

⚡ High impact 🔄 Effort: L 🎯 Confidence: HIGH
F-003 · MAJOR

No verification gates on generated code

Generated code is merged without automated verification against existing tests, lint rules, or conventions. There's no "done criteria" — just subjective review. Regressions compound silently.

📊 Medium impact 🔄 Effort: M 🎯 Confidence: MEDIUM
PLAN-001 — Architecture

A pipeline, not a prompt.

Seven stages. One directional flow. From raw codebase to executable plan — with verification at every step.

Phase I

Recon

Map the repo: stack, conventions, build/test/lint commands. Ingests ADRs, PRDs, DESIGN.md, PRODUCT.md — so decided tradeoffs aren't re-flagged as findings.

Phase II

Audit

Fans out parallel subagents across 9 categories: correctness, security, performance, test coverage, tech debt, dependencies, DX, docs, and direction. Every finding carries file:line evidence.

Phase III

Vet

Subagents over-report, so the advisor re-reads every cited location itself. False positives get dropped, wrong attributions get corrected, rejections get recorded.

Phase IV

Prioritize

Findings land in a table ordered by leverage — impact ÷ effort, weighted by confidence. You pick what becomes plans. The raw data is surfaced, not hidden.

Phase V

Plan

One file per selected finding. Self-contained specs with current-code excerpts, exact steps, and verified commands. Written for the weakest plausible executor.

Phase VI

Execute

Spawns a cheaper executor in an isolated worktree. Reviews results like a tech lead: re-runs done criteria, checks scope compliance, reads diff against intent.

Phase VII

Reconcile

Next session: verify what landed, refresh what drifted, unblock what got stuck. Plans aren't fire-and-forget — they're living documents.

PLAN-002 — Specification

Plans that execute. Not essays.

Every plan has three properties that make it executable by the weakest model — no context-sharing, no "as discussed above," no improvisation.

Self-contained

All context is inlined — exact file paths, current-state code excerpts, repo conventions with an exemplar file, verified commands. No dependencies on prior conversation.

Verification gates

Every step ends with a command and its expected output. Done criteria are machine-checkable. The executor never has to judge whether it succeeded.

Hard boundaries

Explicit out-of-scope lists and STOP conditions — "if X, stop and report" — instead of letting a small model improvise when reality doesn't match the plan.

PHILOSOPHY-001

"You → /improve → plans/ → other agent → done"

Intelligence compounds where it matters — understanding the codebase, judging what's worth doing, writing the spec. The skill never implements anything itself. The plan is the product.

PLAN-003 — Constraints

Hard rules. No exceptions.

improve.sh has strict boundaries. It's an advisor, not an implementer.

Never modifies source code. The only writes go to plans/. Executors edit only in disposable worktrees; merging is always yours.

Never mutates your working tree. Read, search, and read-only analysis only. Your codebase stays untouched.

Never reproduces secret values. Locations and credential types only, rotation always recommended.

Asked to implement? It declines. It points at the plan — or offers /improve execute to dispatch a cheaper executor.
